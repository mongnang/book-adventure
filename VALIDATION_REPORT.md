## 1. OpenAI 속도 최적화와 안전한 시간 기록

- 일반 독서 답변과 인물 대화는 `max_output_tokens: 800`을 사용합니다.
- 활동 1 초안·수정은 800토큰, 최종 이미지 프롬프트는 실제 잘림 재현 뒤 1,600토큰으로 조정했습니다.
- Responses 요청에는 `reasoning.effort: "minimal"`, `text.verbosity: "low"`, `max_output_tokens`를 사용합니다.
- 요청 형식을 실제 Azure 환경에서 확인한 결과 `text.format: {type: "json_object"}`가 현재 배포 모델에서 거부되어 제거했습니다. 대신 JSON 전용 지시와 서버 응답 검증을 유지합니다.
- 원문 근거는 질문·인물과의 관련도 상위 2개만 선택합니다.
- Azure OpenAI, Cosmos DB, 전체 처리 시간을 요청 ID로 연결해 기록합니다.
- 로그에는 단계, API 경로, HTTP 상태, 오류 코드, 비밀정보를 제거한 메시지와 시간만 남깁니다. 프롬프트 원문, 학생 개인정보, API 키와 토큰은 남기지 않습니다.
- `incomplete`, `length`, `max_output_tokens` 종료는 성공으로 오인하지 않고 잘린 응답 오류로 처리합니다.

## 4. 화면 레이아웃과 스크롤

- 첫 제목을 낱말 단위로 묶어 `모험`이 글자 중간에서 갈라지지 않게 했습니다.
- 1440×900, 390×844, 320×640에서 확인했습니다. 320px에서는 `모험` 전체가 다음 줄로 이동하며 글자 단위 분리는 없습니다.
- 활동 2 대화창을 오른쪽 HUD 직전까지 넓혔고 데스크톱 간격은 약 19px였습니다. 390px 모바일에서는 세로 배치되며 겹침과 가로 넘침이 없었습니다.
- 활동 3 인물 목록은 모바일에서 내부 높이 210px, 전체 내용 364px였고 `PageDown`으로 마지막 `성 서방네 처녀`가 보이도록 이동한 뒤 실제 선택했습니다.
- 활동 3 메시지 영역은 독립 스크롤이고 입력창은 계속 사용할 수 있는 위치에 유지됩니다.
- 최신 메시지 4회까지 바닥 이동을 확인했습니다. 실제 휠로 900px 위로 이동한 뒤 새 메시지를 보내자 기존 `scrollTop` 1265가 그대로 유지되어 강제 하단 이동이 없었습니다.
- 모바일 뷰포트와 `touch-action: pan-y` 적용은 확인했지만 실제 터치 장치 제스처는 실행하지 않았습니다.
